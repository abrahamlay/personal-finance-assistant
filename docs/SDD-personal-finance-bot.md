### 4.5.5 Wire Auto-Update Hooks (Critical)

**For graphify + CRG auto-update on every commit and branch switch**, follow the **post-commit and post-checkout hook setup** in the [Dev.to article § "Auto-Update on Commit"](https://dev.to/mir_mursalin_ankur/graphify-code-review-graph-build-a-self-updating-knowledge-graph-for-claude-code-and-other-ai-j1m#autoupdate-on-commit-the-full-picture).

**TL;DR:**
```bash
# 1. Copy full post-commit hook (with _resources_ok guard) to .git/hooks/post-commit
# 2. Copy full post-checkout hook to .git/hooks/post-commit
# 3. Make executable
chmod +x .git/hooks/post-commit .git/hooks/post-checkout

# 4. Test
git add . && git commit -m "test"
# Should see: "[graphify hook] launching background rebuild"
# and: "[code-review-graph] Commit detected - launching background update"
```

**Why resource guards matter:** Without them, rapid commits (amend chains, rebase) spawn concurrent graphify/CRG processes that spike CPU to 12+ load average and saturate swap. The `_resources_ok()` check in the article prevents this.

**Checkpoint:** Git log shows commits; graph files in `.code-review-graph/` and `graphify-out/` have recent mtimes (`stat -f %Sm <file>` on macOS).

### 4.5.6 Set Up Smart-Grep Hook (Optional but Recommended)

The smart-grep hook intercepts `grep`, `rg`, `find` commands and routes them through the knowledge graph first — preventing 1300–2500 token waste on file reads.

**Setup:**
```bash
# Create hook script (from Dev.to article § "Smart Grep Hook")
mkdir -p ~/.claude/scripts
# Copy the full smart-grep-hook.sh from the article into ~/.claude/scripts/smart-grep-hook.sh
chmod +x ~/.claude/scripts/smart-grep-hook.sh

# Wire into Claude Code's PreToolUse hook
cat >> .claude/settings.local.json <<'EOF'
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "bash ~/.claude/scripts/smart-grep-hook.sh",
          "timeout": 6
        }]
      }
    ]
  }
}
EOF
```

**What it does:**
- First grep this session on a graph hit → shows answer + allows grep (teaches the agent)
- Subsequent greps on a graph hit → denies with answer inline (no retry loop)
- No graph hit → allows grep silently
- Graph exists but not for this query → suggests MCP tool for next time

**Cost impact:** Prevents ~1300–2500 token waste per graph-answerable query. On 36 graph tool calls/session, saves ~90,000 tokens.

**Reference:** [Dev.to § "Smart Grep Hook"](https://dev.to/mir_mursalin_ankur/graphify-code-review-graph-build-a-self-updating-knowledge-graph-for-claude-code-and-other-ai-j1m#enforcing-graphfirst-the-smart-grep-hook)

**Checkpoint:** Grep returns graph context before executing (test with `grep -rn "function_name" src/`).

### 4.5.9 CRG Stop Hook (AI-Driven Updates)

When Claude Code finishes a turn, CRG incremental update runs in background (~0.425s, PID-guarded).

**Already in `.claude/settings.example.json` from `code-review-graph install`.** Copy to `.claude/settings.local.json`:

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [{
          "type": "command",
          "command": "command -v code-review-graph >/dev/null 2>&1 && [ -d .code-review-graph ] && { PF=/tmp/crg-claude.pid; if [ -f \"$PF\" ] && kill -0 \"$(cat \"$PF\")\" 2>/dev/null; then true; else { code-review-graph update --skip-flows 2>/dev/null && nohup code-review-graph embed >/dev/null 2>&1 & } & echo $! > \"$PF\"; fi; } || true",
          "timeout": 5
        }]
      }
    ]
  }
}
```

**Why Stop, not PostToolUse?** PostToolUse fires per file edit (10 edits = 10 triggers). Stop fires once per turn. Prevents process pile-up and CPU spike.

**Checkpoint:** Claude Code finishes editing; no background CRG processes accumulate.

## Knowledge Graph Query Patterns (Day-to-Day)

| Question | Tool | Cost |
|----------|------|------|
| "Where is `createThread` defined?" | `semantic_search_nodes_tool(query=createThread)` | ~115 tok |
| "Who calls `createThread`?" | `query_graph_tool(pattern=callers_of, target=createThread)` | ~80 tok |
| "What breaks if I change server.ts?" | `get_impact_radius_tool(changed_files=[server.ts])` | ~70 tok |
| "Code review context for PR?" | `get_review_context_tool(changed_files=[...])` | ~111 tok |
| "Module clusters / architecture?" | `list_communities_tool()` | ~170 tok |
| "Shortest path A → B?" | `graphify path A B --graph graphify-out/graph.json` | ~200 tok |
| "Explore payment flow?" | `graphify query "payment" --graph graphify-out/graph.json` | ~1,500 tok (BFS) |
| "String search (can't graph)?" | `grep -rn "error_handler" src/ # --graph-tried` | ~550 tok + follow-up reads |

**Smart routing (what the hooks do):**
1. Agent asks "where is X?" → hook → `semantic_search_nodes_tool` (~115 tok, done)
2. Agent reaches for grep → `smart-grep-hook` → checks graph first → shows result inline, agent skips grep (saves 1,300 tok)
3. Session starts → `SessionStart` hook → shows cheatsheet once (150 tok, prevents reflexive grep spam all session)

### 7.6 Knowledge Graph Edge Cases

#### 7.6.1 CRG Incremental Update Lags After Large Refactor
**Symptom:** `code-review-graph status` shows old node/edge counts after touching 150+ files
**Root Cause:** Post-commit hook detected >5 files changed, triggered full `build` instead of incremental `update`; build takes 1–3 min in background
**Fix:**
```bash
# Check if build is still running
pgrep -af 'code-review-graph build'
# If yes, wait or check log:
tail -f ~/.cache/code-review-graph-update.log

# If build failed, rebuild manually
code-review-graph build
```

**Prevention:** This is correct behavior — large refactors need full rebuilds. The `.code-review-graph/` directory is gitignored, so every `git checkout` on a branch with different file counts retriggers the smart rebuild logic.

---

#### 7.6.2 graphify Update Keeps Timing Out
**Symptom:** `~/.cache/graphify-rebuild.log` shows timeout after 300s; graph not updating
**Root Cause:** Large monorepo (10K+ files); graphify update exceeds 5-minute timeout
**Fix:**
```bash
# Increase timeout in .git/hooks/post-commit from `timeout 300` to `timeout 600`
# Then test manually with --force:
graphify update . --force

# Check SHA256 cache health (if broken, rebuilds unnecessarily)
ls -lh graphify-out/cache/ | head -10
# If thousands of entries, consider `graphify update . --skip-cache`
```

---

#### 7.6.3 Smart-Grep Hook Not Firing / Grep Proceeds Without Graph Intercept
**Symptom:** Agent runs `grep -rn "TODO" src/` without graph context shown
**Root Cause:** PreToolUse hook not in `.claude/settings.local.json` or smart-grep-hook.sh has permission error
**Fix:**
```bash
# Check settings.local.json has PreToolUse hook:
grep -A3 '"PreToolUse"' .claude/settings.local.json

# Check script is executable and in right location
ls -la ~/.claude/scripts/smart-grep-hook.sh
chmod +x ~/.claude/scripts/smart-grep-hook.sh

# Test hook directly
echo '{"tool_input":{"command":"grep -rn PortfolioViewModel src/"}}' | \
  bash ~/.claude/scripts/smart-grep-hook.sh

# Should output: graph query result or pass-through decision
```

---

#### 7.6.4 CRG Embeddings Not Updating (Semantic Search Returns No Results)
**Symptom:** `semantic_search_nodes_tool(query=PortfolioViewModel)` returns 0 results despite symbol existing
**Root Cause:** Embeddings not generated (Stop hook only runs `update --skip-flows`, not `embed`); or embeddings cache stale
**Fix:**
```bash
# Force embed
code-review-graph embed --force

# Check embed status
code-review-graph status | grep -i embed
# Should show: "Embeddings: enabled, 5780 nodes indexed"

# Verify Stop hook includes embed command (should be there)
grep -o "code-review-graph embed" .claude/settings.local.json
```

#### 7.6.5 graphify Query BFS Timeout (>10s, returns no result)
**Symptom:** `graphify query "payment system" --graph graphify-out/graph.json` hangs or times out
**Root Cause:** BFS depth=2 on large graph hits memory/time limits (~1,500 token output, ~10s runtime)
**Fix:**
```bash

```

## One System-Level Risk Not Mitigated

**The risk:** Post-commit and post-checkout hooks run concurrent background processes. On rapid-fire commits (amend chains, interactive rebase), multiple graphify/CRG rebuilds stack up — observed: 3 stuck processes at 65–73% CPU each, load average 12+, RAM saturated, machine unresponsive.

**Mitigation in this plan:**
- `_resources_ok()` function checks CPU (≤50% of logical cores) and memory (≥2GB free) before spawning
- `pgrep` guard prevents duplicate processes
- `timeout 300` caps max runtime to 5 minutes
- graphify runs as detached `nohup` (doesn't block git operations)

**Residual gap:** On machines with 2 logical cores + <4GB RAM (edge case), even the guards may not prevent slowdown. If you see `load average > 8` during a session, immediately run:
```bash
killall graphify graphifyy code-review-graph  # kill background rebuilds
pgrep -af 'graphify|code-review' | wc -l      # verify killed
```

Then re-run `graphify update .` and `code-review-graph build` after the heavy work passes.
